About
I think first.
Then I build.

AI/ML engineering student from Coimbatore, India. I don't prototype — I build things I'd actually run, maintain, and trust in production. That means real architecture decisions, real failure handling, real persistence. Not a demo. A system.

The way I approach a problem: understand the domain cold before writing the first line. I've gone deep into LLM orchestration, database engineering, voice systems, browser automation, and full-stack development — not to check boxes, but because you can't architect something you don't fully understand.

I hold myself to a single standard — does it still work when things go wrong? Good systems don't just handle the happy path. They degrade gracefully, recover cleanly, and don't need babysitting.

That thinking shows up in every system I've built. Here's what that looks like in practice.

Live Demo · DIABLO
"Wakes on command. Works without rest."

Built entirely from scratch as a local AI operating system. Fourteen layers, 55+ tools across 9 namespaces — wake word detection via "Hey DIABLO", screen vision through llama-4-scout, full voice I/O with Edge TTS, Playwright browser automation with a persistent Chrome profile, SQLite memory across sessions, smart VAD with silence detection, a daily rate limiter, and self-correction when a tool call fails.

Nothing calls out to a cloud API unless you point it somewhere. It runs on the machine and stays on the machine.

"3.7 million rows. One question. Instant answer."

Full-stack financial platform with 3.7M+ rows of S&P 500, crypto, and macro data in TimescaleDB. Ask a question in plain English — Groq handles the translation, and the precomputed indicator pipelines return a real answer in under a second.

Live Demo · AI Career Copilot
"Understands your resume like a senior recruiter."

Scores resumes across four weighted dimensions — Skills (35%), Experience (30%), Keywords (20%), Education (15%). Cross-checks resume claims against your actual GitHub repos and generates role-specific interview prep with full STAR guidance.

Coding challenge engine across 6 languages with AI code review. Multi-resume ranking for recruiters. Groq API key rotation keeps throughput steady. The weighting exists because not everything on a resume matters equally.

Live Demo · AI Code Intelligence
"Reasons about code like a senior engineer would."

12-stage analysis pipeline for entire codebases. Semantic search using SentenceTransformers + ChromaDB. Dependency graphs via NetworkX — see exactly how every file connects to the rest. Natural language Q&A over code, answered locally through Ollama/Llama3.

The Arsenal
Tools I build with.
And why.
AI · LLM
Groq API
Why Fastest LLM inference. Used across DIABLO, Career Copilot, and Market Terminal.
Tradeoff Rate limits at scale. Key rotation needed for heavy use.
Whisper STT
Why Open-source speech-to-text. Runs locally. Powers DIABLO's voice input.
Tradeoff RAM-heavy. Needs silence detection to avoid wasted inference.
Edge TTS
Why Natural voice output, free, no key needed. DIABLO's spoken replies.
Tradeoff Needs internet. Not fully offline.
Database
PostgreSQL
Why Production-grade relational DB. Powers Market Intelligence Terminal.
Tradeoff Needs a running server. Overkill for small projects.
SQLite
Why Zero setup, file-based. DIABLO's persistent memory lives here.
Tradeoff Not great for concurrent writes or large-scale production.
ChromaDB
Why Vector DB for semantic search. Backbone of Code Intelligence RAG.
Tradeoff In-memory by default. Persistence needs explicit setup.
Backend
Python
Why Primary language. Best ecosystem for AI, automation, and data engineering.
Tradeoff Slower than compiled languages. GIL limits true multithreading.
FastAPI
Why Fast async Python API framework. Backend for all 4 projects.
Tradeoff Less structured than Django. You define the architecture yourself.
Playwright
Why Full browser automation with persistent Chrome profile. DIABLO's browser control.
Tradeoff Heavy. Needs a browser process running in background.
Frontend
Next.js 14
Why React framework with routing and API routes built in. Career Copilot and Market Terminal.
Tradeoff More overhead than plain React for simple UIs.
React
Why Component-based UI. Used across all frontend projects.
Tradeoff State management can get complex at scale.
Tailwind CSS
Why Utility-first CSS. Fast to build with, consistent design out of the box.
Tradeoff HTML gets verbose. Custom designs need extra config.
